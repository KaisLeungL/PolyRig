# PolyRig Pack Authoring Contract

This contract expands the lean `SKILL.md`. Follow it when creating or updating a
PolyRig pack.

## Responsibility

`polyrig-pack-author` produces and maintains packs. `/polyrig` consumes packs and
copies their knowledge into target projects. Do not mix those roles.

## Write Roots

Default to user-level packs:

```text
~/.polyrig/packs/<type>/<short-name>/
```

Only write elsewhere after the user explicitly chooses it:

- Project pack: `<target>/.polyrig/packs/<type>/<short-name>/`
- Builtin pack: `<POLYRIG_ROOT>/packs/<type>/<short-name>/`

If the selected path exists, treat the work as an update. Read the existing
`pack.yaml`, `knowledge/`, `deps.yaml`, `verify.md`, and `references/sources.md`
before editing. Preserve useful content and make minimal changes.

## Create vs Update

For create:

1. Copy the template shape from `assets/pack-template/`.
2. Replace placeholder identity, dates, summary, metadata, evidence rows, and
   knowledge sections.
3. Add `knowledge/per-stack/<stack>.md` files when a domain pack declares
   `stacks`.

For update:

1. Identify the reason for the update: new use case, source refresh, evidence
   repair, dependency lookup change, or compatibility metadata change.
2. Keep existing stable decisions unless the new evidence contradicts them.
3. Bump `last_reviewed` when a human review is complete.
4. Keep evidence ids stable when possible; add new ids for new claims.

## Six-Phase Workflow

### 1. Pack Identity

Collect:

- mode: create or update
- type: `stack` or `domain`
- id: `stack/<short-name>` or `domain/<short-name>`
- version: semver for pack content
- target root: user, project, or builtin
- trust posture appropriate for that root

The directory path must agree with `id` and `type`.

### 2. Use Cases

Capture:

- decisions future agents should make from the pack
- tasks that should trigger this pack
- verification scenarios the pack must support
- what a competent model would otherwise miss

Skip generic textbook material that agents already know.

### 3. Boundaries

Capture:

- covered scope
- explicit non-goals
- stack compatibility for domain packs
- `requires`, `conflicts`, and `provides`
- whether scripts are requested

Scripts are not generated by default. If the user asks for scripts, they must be
deterministic helpers with tests and must not be required for the pack to be
useful.

Before recording any `requires` entry, run the **dependency pre-scan** (see
"Scan the registry before introducing a dependency" below) so a typo or dangling
reference is caught early.

### 4. Source Plan

Before web search or local reading, tell the user the planned source scope.

Preferred source types:

- official documentation
- standards documents
- security, privacy, and reliability documentation
- vendor documentation
- user-provided local files or pasted material

Boundary:

- Do not treat forums, marketing blogs, SEO articles, social media, or anonymous
  examples as authoritative.
- Do not write current "latest" versions or volatile API details into prose.
- Put volatile facts in `deps.yaml` as lookup strategies.
- If a claim lacks a source, mark it `inferred` or `unverified` in the Evidence
  Matrix and avoid using it for strong rules.

## Pack Files

Required shape:

```text
pack.yaml
knowledge/
  overview.md
  pitfalls.md
references/
  sources.md
verify.md
```

Add `deps.yaml` when the pack needs volatile dependency, API surface, package, or
version lookups. If `deps.yaml` exists, every dependency entry must include
`evidence: [...]`.

`knowledge/overview.md` contains slow-changing decisions, principles, defaults,
and decision trees.

`knowledge/pitfalls.md` contains traps, red lines, failure modes, and practices
to avoid.

`deps.yaml` contains volatile lookup strategies. Every dependency entry needs an
`evidence` array.

`verify.md` contains concrete automated/manual completion checks. Every tool it
invokes through a package runner (`uv run`, `uvx`, `poetry run`, `pipx run`,
`pdm run`, `npx`, `pnpm dlx`/`exec`, `yarn dlx`, `bunx`) must have a `deps.yaml`
entry so assembly can resolve its version into the target project's
`deps.resolved.md` — `validate-pack.mjs` enforces this. System toolchain
commands (`xcodebuild`, `./gradlew`, `adb`, …) are exempt.

`references/sources.md` contains the Evidence Matrix.

## Evidence Matrix

`references/sources.md` must contain:

```md
## Evidence Matrix

| id | claim | status | source_type | urls | applies_to | volatility | notes |
|---|---|---|---|---|---|---|---|
```

Allowed `status` values:

- `source-backed`
- `user-provided`
- `inferred`
- `unverified`

Allowed `volatility` values:

- `low`
- `medium`
- `high`

Evidence ids use `E001`, `E002`, and so on. Use inline markers in pack prose:

```md
Do not commit credentials, signing keys, or private tokens. [Evidence: E001]
```

Strong rules and red lines need source-backed or user-provided evidence. Do not
support strong rules only with `inferred` evidence, and never support them with
`unverified` evidence.

## Draft / Ready Gate

Always write a draft before declaring ready.

Start the independent verification gate. The implementing agent must not verify
its own pack in the same context when subagents or fresh independent sessions are
available.

Assign an independent verification context to run the validator:

```bash
node scripts/validate-pack.mjs <pack-dir>
```

Run two reviews using `references/review-prompts.md`:

1. Protocol / structure reviewer
2. Content / safety reviewer

The verification and review gate is an automated phase of
`polyrig-pack-author`, not a handoff to the user. After writing the draft, the
acting agent must start an independent validation context and then the two
reviews when independent contexts are available, preferably as subagents. A
self-validation or self-review in the implementation context does not satisfy
the gate.

Only defer the gate when no independent validation/reviewer mechanism is
available or the user explicitly asks to stop before review. In that case,
report the pack as `draft written`, say exactly why the gate was not run, and
give the command and two fixed prompts needed to resume. Do not describe the pack
as `ready`, `verified`, or complete.

Ready criteria:

- validator exits 0 in an independent verification context
- protocol / structure reviewer has no blocking issues
- content / safety reviewer has no blocking issues

If any criterion fails, report the pack as draft only, list blocking issues, and
recommend the next fixes.

`doctor.mjs` remains a deterministic health check. It may validate pack shape
and installation links, but it does not replace the two-review ready gate.

## Scan the Registry Before Introducing a Dependency

This applies to **both** a pack's `requires` and a group's `requires`. Before
recording a dependency on another pack id, verify that id actually resolves,
using this fallback order:

1. **Local discovery roots.** Run `node scripts/build-pack-index.mjs` and check
   whether the id appears (builtin, user, and project roots). If present, the
   dependency is verified.
2. **Registry.** If the id is not local and `POLYRIG_REGISTRY_URL` is set and
   reachable, check whether the registry publishes that pack id. If present, the
   dependency is verified.
3. **Neither resolves.** Warn the user with the exact text
   `疑似无效引入，请确认`, record the dependency as `unverified`, but do **not**
   block authoring.

This author-side scan is an **early hint** — it catches typos and dangling
references before publish. It is not the authoritative gate. The final defense
is the registry's joint-validate at publish time, which validates every member
and external reference together in one isolated staging area.

## Group Authoring (`groups/<name>/group.yaml`)

A **pack group** is a versioned, reference-style manifest that bundles member
packs by id + pinned version. It is the install and publish unit; the member
packs stay atomic and are not modified by grouping them. There is no "public
component" concept — a pack a group depends on from outside is just an ordinary
pack referenced through the group's `requires`. group.yaml is validated against
`schemas/group.schema.json` and the six invariants in `scripts/validate-group.mjs`.

### Write root

```text
<POLYRIG_ROOT>/groups/<name>/group.yaml     # builtin group
```

Groups follow the same root-choice discipline as packs (default user-level;
project or builtin only after an explicit choice). The directory `<name>` must
agree with the `id` (`group/<name>`).

### group.yaml fields

- `id`: required, `^group/[a-z0-9-]+$`.
- `version`: required semver. It exactly locks every reference below; **any**
  member content change requires a new group version.
- `last_reviewed`: required, `YYYY-MM-DD`.
- `summary`: required, non-empty.
- `members`: required, non-empty. Each item is `{ id, version }` where `id` is a
  `<stack|domain>/<name>` pack and `version` is the exact pinned semver.
- `requires`: optional (default `[]`). Each item is `{ id, version }` pointing at
  an **out-of-group** ordinary pack, also exactly pinned. An id here must not
  also appear in `members`.
- No Evidence Matrix (a group carries no knowledge; evidence stays in each
  member pack's `references/sources.md`).

### YAML caveat — block style only

PolyRig's YAML subset parser does **not** support flow maps (`{...}`). Always
write members and requires in block style:

```yaml
id: group/auth
version: 0.1.0
last_reviewed: 2026-07-06
summary: OAuth/OIDC sign-in suite — shared auth-core plus Google and GitHub providers
members:
  - id: domain/auth-core
    version: 0.1.0
  - id: domain/auth-google
    version: 0.1.0
  - id: domain/auth-github
    version: 0.1.0
requires: []
```

### Authoring subflow

1. **Group identity** — create or update; confirm id, summary, target root.
2. **Members** — choose member packs; for each, **auto-read its current
   `pack.yaml` version** and pin exactly that version to generate the lock.
   Members must be unique.
3. **External dependencies** — declare any out-of-group `requires`, pinned to an
   exact version; run the dependency pre-scan on each. An id in `requires` must
   never also be a member.
4. **Validation gate** — run the group validator (below) as the local
   pre-publish gate.

### Group validation gate

```bash
node scripts/validate-group.mjs <path-to-group.yaml>
```

The validator enforces the six invariants:

1. **Dependency closure** — every member's direct `requires` target must be
   another member or a declared group-level `requires` (checked one level deep).
2. **Members exist and versions match** — each `members[].id` resolves and its
   `pack.yaml` version equals the pinned version; group-level `requires` too.
3. **No duplicate / valid ids** — member ids are unique and well-formed.
4. **Members vs external deps are disjoint** — no id is both a member and a
   `requires` entry.
5. **Members do not conflict** — no member's `pack.yaml` `conflicts` names a
   sibling member.
6. **Acyclic `requires` graph** — the (members + external deps) `requires` graph
   topologically sorts with no cycle; that order also drives install closure
   ordering.

Report the group `ready` only when the validator exits 0. Otherwise report
`draft written` and list the violations. As with packs, the registry re-runs a
pinned joint-validate at publish — the local run is the early gate, not the final
authority.
